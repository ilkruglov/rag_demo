# Universal RAG Demo Engine

Инженерное описание проекта: что это за система, на каком стеке она построена и как устроен ее RAG-пайплайн.

## Что это за проект

`rag_chatbot` - это локально запускаемый RAG-сервис с веб-интерфейсом и API для работы с собственной базой документов.

Система умеет:

- принимать `.docx` и `.pdf` документы;
- автоматически переиндексировать базу после загрузки или удаления файлов;
- искать релевантные фрагменты по гибридной схеме;
- генерировать ответ LLM-моделью на основе найденного контекста;
- возвращать ответ вместе с источниками.

Проект рассчитан на сценарий "загрузи документы -> задай вопрос -> получи ответ с цитируемыми источниками".

## Основная идея

Внутри это не "просто чат с файлами", а связка из нескольких слоев:

- Streamlit UI для работы пользователя;
- FastAPI backend для операций с документами и чатом;
- ingestion-конвейер для разбора, нормализации и индексирования документов;
- hybrid retrieval: векторный поиск + BM25;
- reranking и доменные бустеры;
- Groq LLM для финальной генерации ответа.

## Технический стек

| Слой | Технологии | Роль |
| --- | --- | --- |
| UI | Streamlit, `httpx` | Интерфейс чата, документов и настроек |
| API | FastAPI, uvicorn | HTTP API для чата и управления KB |
| RAG orchestration | LlamaIndex | Query engine, retrievers, response synthesis |
| LLM | Groq | Генерация ответа по найденному контексту |
| Embeddings | HuggingFace, `intfloat/multilingual-e5-large` | Векторизация чанков и запросов |
| Vector store | Qdrant | Хранение и поиск по эмбеддингам |
| Keyword retrieval | `rank-bm25`, `pymorphy3` | Лексический поиск с русской лемматизацией |
| Document parsing | `python-docx`, `pypdf`, кастомный table parser | Извлечение текста и таблиц из документов |
| Runtime config | Pydantic Settings, `.env`, `runtime_settings.json` | Управление параметрами без правки кода |

## Как устроен RAG

### 1. Ingestion

После загрузки файлов backend запускает полную переиндексацию.

Этапы:

1. Документы читаются из `data/raw`.
2. `.docx` разбираются с сохранением порядка абзацев и таблиц.
3. Таблицы проходят через специальный parser, который пытается извлечь:
   - коды операций;
   - коды форм;
   - тип содержимого таблицы;
   - дополнительные структурные признаки.
4. Документы нормализуются:
   - удаляются пустые строки;
   - устраняются дубликаты;
   - добавляются базовые metadata.
5. Текст режется на чанки через `SentenceSplitter`.
6. Для чанков вычисляются эмбеддинги.
7. Параллельно строится BM25-индекс по тем же node'ам.
8. Индексы сохраняются в локальные хранилища.

### 2. Metadata enrichment

Во время индексации система пытается сделать чанки не только текстовыми, но и структурно осмысленными.

Для чанков сохраняются такие поля:

- `document_title`
- `document_filename`
- `document_category`
- `page`
- `section_title`
- `chunk_id`
- `operation_codes`
- `form_codes`
- `content_type`
- `operation_direction`

Это важно, потому что retrieval и reranking затем используют эти metadata как сигналы релевантности.

### 3. Hybrid retrieval

Поиск строится по двум каналам одновременно:

- dense retrieval через Qdrant;
- sparse retrieval через BM25.

Потом результаты объединяются через `QueryFusionRetriever` в режиме reciprocal rank fusion.

Текущая идея ранжирования:

- vector retrieval хорошо находит семантически похожие фрагменты;
- BM25 хорошо работает на точных терминах, кодах операций, названиях форм и юридических формулировках;
- fusion позволяет не проваливаться ни на "точных" запросах, ни на "смысловых".

### 4. Query expansion

Если включены доменные правила, система пытается расширить пользовательский запрос:

- определяет направление операции;
- подмешивает связанные коды операций;
- подмешивает коды форм.

Это особенно полезно для доменных запросов, где пользователь пишет не точный код, а смысл операции.

### 5. Reranking и бустеры

После retrieval система применяет несколько постпроцессоров:

- `OperationCodeBooster` - усиливает документы с совпадающими кодами операций;
- `SemanticDirectionBooster` - усиливает фрагменты с правильным направлением операции;
- `SectionTitleBooster` - усиливает совпадения по заголовкам разделов;
- reranker - либо легкий keyword-overlap, либо cross-encoder.

В результате на вход LLM уходит уже не "сырой top-k", а более плотный и предметно точный набор источников.

### 6. Ответ LLM

LLM получает:

- системный промпт;
- собранный контекст;
- исходный вопрос пользователя.

После генерации ответ дополнительно очищается:

- убираются служебные `<think>`-фрагменты;
- убираются нежелательные дополнительные секции;
- формируется список источников из `source_nodes`.

Итог API:

- `answer`
- `sources[]`

## Что делает проект хорошо

### Сильные стороны

- Гибридный retrieval вместо одного вида поиска.
- Хорошая работа с русскоязычными терминами за счет BM25 и лемматизации.
- Поддержка таблиц как полноценных знаний, а не шумного текста.
- Богатые metadata для explainability и качества retrieval.
- Настройки модели, промпта и retrieval меняются без редактирования кода.
- Локальное хранение индексов и документов, без обязательной внешней базы.

### Где проект особенно силен

Архитектура особенно хорошо подходит для:

- внутренних knowledge base;
- регламентов и инструкций;
- юридических и операционных документов;
- предметных корпусов, где важны коды, формы, номера операций и структура разделов.

## Важные технические нюансы

### Настройки UI и backend

UI сохраняет runtime-настройки напрямую в `config/runtime_settings.json`.

Это удобно, но есть нюанс:

- backend и UI запущены как разные процессы;
- настройки на backend читаются через кешируемый `get_settings()`;
- поэтому часть изменений может потребовать сброса кеша или перезапуска backend.

### Синхронизация чата и переиндексации

В backend есть защита от гонок:

- во время ingestion новые chat-запросы временно блокируются;
- операции с документами выполняются под lock;
- после переиндексации кеш query engine сбрасывается.

Это снижает риск ответа по устаревшему или частично обновленному индексу.

## Короткий итог

`rag_chatbot` - это гибридная документная QA-система с локальным vector store, BM25-поиском, LLM-генерацией и доменными усилителями качества retrieval.

Ее главная ценность не только в "ответах по документам", а в том, что она:

- умеет работать со структурой документов;
- учитывает точные термины, коды и формы;
- дает управляемый retrieval;
- возвращает прозрачные источники;
- остается достаточно простой для локального запуска и доработки.
